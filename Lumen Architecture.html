01

Two guiding rules

Every architectural decision in this document follows from these two rules. When in doubt, come back to them.

Rule 1

The engine is sacred

imaging.js, analysis.js, crop.js and the rest of the imaging engine do not change. They operate on ImageData buffers passed to them — they know nothing about storage, sync or UI state. Keep it that way forever.

Rule 2

Everything goes through a provider

The UI never touches indexedDB directly. All persistence flows through IStorageProvider. Swapping local storage for cloud sync means changing one line — the provider constructor.

03

Architecture layers

Three layers, clearly separated. The imaging engine sits outside all three — it's a pure function library that transforms pixels.

UI Layer

js/ui/
app.js, panels.js, crop.js, export.js, dashboard.js. Reads the library from window.Store.library. Writes edits via Store.saveEdit(entry). Never calls IndexedDB directly. Subscribes to store events to re-render when data changes.

Store LumenStore

js/core/store.js
Central state. Holds the in-memory library[] (working entries the engine edits). On boot, loads from provider. On edit/import/delete, updates memory + calls provider. Emits photosChanged / selectionChanged events.

IStorageProvider

interface
The contract. Declares every method the app needs from a storage backend. The Store calls this interface; it never knows which backend is behind it.
LocalProvider Phase 0–3 · now
  • IndexedDB via LumenDB helpers
  • Photos, edits, albums, presets in object stores
  • Original blobs in a separate blobs store
  • Sync methods return no-ops
CloudProvider Phase 4 · future
  • Same interface, different backend
  • Reads/writes Supabase Postgres
  • Thumbnails to Supabase Storage
  • Sync cursor + outbox implemented

The imaging engine floats above all of this

Imaging.process(imageData, params, out) is a pure function — it takes pixels in, puts pixels out. It has no storage, no state, no network. It belongs to no layer. This is what makes it possible to ship Phase 0 without touching the engine at all.

06

Privacy guarantees

Privacy is architectural — these are structural properties of the code, not promises in a policy document.

All pixel processing is in-browser

The imaging engine runs on ImageData buffers in the main thread. No pixel data is ever serialised to a network request.

Original files never leave the device without your action

The blobs object store is never touched by sync code. LocalProvider.storeOriginal() and loadOriginal() are the only callers — both local.

No analytics, no telemetry, no external scripts

The only external requests are the Babel/React CDN scripts at load time (dev only). In production these should be locally bundled. No tracking pixels, no beacons.

GPS scrubbing in the export pipeline

The existing export dialog already strips GPS from EXIF before writing the file. This is preserved in the refactored version.

App works fully without signing in

The store is initialised with new LocalProvider() regardless of auth status. An account is only required to start syncing — it's never required to edit.

Browser storage limits apply

IndexedDB quota varies by browser and OS (typically 10–50% of free disk). For a 10,000-photo library storing thumbnails + blobs, request persistent storage on first use: navigator.storage.persist().

08

Cloud migration path

When you're ready to add sync (Phase 4), the change is surgical. The editor, the UI, the engine — nothing changes. Only the provider constructor is swapped.

Phase 0–3 · Local-only
// In app.js, on boot:
const provider = new LocalProvider();
window.Store = new LumenStore(provider);
await window.Store.init();

// Everything else is unchanged.
// Photos stay on this device.
Phase 4 · Cloud sync enabled
// In app.js, on boot:
const provider = user.isLoggedIn
  ? new CloudProvider(supabaseClient)
  : new LocalProvider();
window.Store = new LumenStore(provider);
await window.Store.init();
// ↑ This is the entire change.

CloudProvider implements IStorageProvider

CloudProvider will be a new class in js/core/cloud-provider.js, extending IStorageProvider. It wraps the Supabase JS client. The storeOriginal / loadOriginal methods still point to IndexedDB — originals are never uploaded without a separate explicit action.

09

Build checklist

Ordered so each step is self-contained and testable before the next begins. Phase 0 is entirely groundwork — nothing in the UI changes yet.

Phase 0 — Foundation (no UI changes)

1

Create folder structure

Make js/core/, js/engine/, js/ui/, icons/. Move existing engine files to js/engine/, UI files to js/ui/. No content changes yet.

2

Add js/core/db.js

IndexedDB open/upgrade. Seven object stores with correct indexes. Version-gated migrations.

3

Add js/core/storage-provider.js

IStorageProvider class with all methods throwing _ni(). This is the contract.

4

Add js/core/local-provider.js

LocalProvider extends IStorageProvider. Full IndexedDB implementation of all methods.

5

Add js/core/store.js

LumenStore constructor takes a provider. Implements init(), addPhoto(), saveEdit(), deletePhoto(), select(), on()/emit().

6

Generate icons + add manifest.json

Generate 192px and 512px PNG icons from canvas. Write manifest.json with file_handlers and share_target.

7

Add sw.js + register in index.html

Cache-first service worker. Register in index.html. Verify app loads offline in Chrome DevTools → Application → Service Workers.

8

Update index.html script order

Add core scripts before engine scripts. Add PWA meta tags. Add store initialisation to app.js.

9

Test install + persistence

Install to desktop from Chrome. Add a photo, close, reopen — it should still be there. Clear site data, reopen — app loads offline.

Phase 1 — Wire app.js to the store

A

Replace library[] with Store.library

In app.js: remove const library = [], initialise Store on load, subscribe to photosChanged to re-render the filmstrip.

B

Persist edits on commit

After every commitEdit(), call Store.saveEdit(entry). Edits now survive page refresh.

C

Persist on import

In loadFiles(), call Store.addPhoto(file, img, exif) instead of building the entry inline. Store saves the blob + metadata to IndexedDB.

D

Persist presets

Save custom presets via Store.savePreset(). Load on boot. Built-in presets stay hardcoded in presets.js.

E

Persist app settings

Save default strength via Store.saveSetting('strength', value). Load on boot with fallback.

Ready to start?

All four core files — db.js, storage-provider.js, local-provider.js and store.js — are already written and waiting in js/core/, along with manifest.json and sw.js. Phase 0 checklist steps 2–6 are done. Begin at step 7: register the service worker and run the install test.
