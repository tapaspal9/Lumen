00

Executive summary

What you have, what we're building, and the one decision that makes "sync everything, free, 10,000 photos" actually possible.

Today

A polished single-page editor. A real canvas imaging engine, auto-analysis, presets, EXIF, crop and web export — but the library lives only in memory. Refresh and it's gone. Desktop layout only.

The goal

A personal Lightroom for travel: import → analyze → suggest → preset → fine-tune → sync → export for your website. Installable on every device, touch + mouse, works offline.

The approach

Local-first. Every device holds its own copy and works fully offline. A thin cloud layer syncs the catalog — never forcing you to upload every original.

The honest tradeoff up front

You asked for true sync and free / local-only. Both can be true — but not for the raw photo bytes. 10,000 travel photos is roughly 150–400 GB; nothing free syncs that. The fix is to sync the part that's tiny and precious (your edits, presets, albums, ratings — a few MB) and leave the giant original files local-first, with optional backup to storage you already own. That keeps the everyday experience free and instant.

01

The core principle: catalog vs. originals

This single idea — borrowed from how Lightroom actually works — resolves the cost/sync tension and shapes every other decision in this document. Lumen edits are already non-destructive (just numbers: exposure +12, a crop rectangle, a preset id), which is exactly what makes it possible.

The Catalog — syncs everywhere

Your decisions about photos

≈ 2–5 KB per photo · ~30 MB for 10,000
  • Edit parameters (exposure, color, crop, straighten)
  • Presets, applied preset + intensity
  • Albums, ordering, ratings, flags, keywords
  • Analysis results & scores (cached)
  • EXIF / camera metadata, capture date, GPS flag
  • Small thumbnails for browsing (~15 KB each)

Tiny, text-like, conflict-friendly → syncs free, in real time.

The Originals — stay put

The raw pixel files

≈ 4–40 MB per photo · 150–400 GB for 10,000
  • Full-resolution JPEG / HEIC / RAW files
  • Held on the device where you imported them
  • Stored on disk via the browser (OPFS / IndexedDB)
  • Optional backup to storage you already own
  • Fetched on demand when you need full-res
  • Never required to be uploaded to use the app

Huge, expensive to move → local-first, opt-in to cloud.

What this gets you

Open Lumen on your phone mid-trip and every album, edit and preset from your laptop is already there. Browse 10,000 photos by thumbnail anywhere. Pull the full original only when you actually need it.

The one limitation to accept

A full-resolution original is only on the device(s) that have its bytes. On a device that doesn't, you'll see the thumbnail + all metadata and can re-export once the original is fetched. Turn on optional cloud backup to erase even that gap.

02

Recommended architecture

Chosen for maintainability and simplicity — the fewest moving parts that still deliver real sync. Your existing imaging engine is kept exactly as-is; we build a data layer around it rather than rewriting it.

App shell
Keep today's vanilla-JS engine (imaging, analysis, presets, crop, export). Add a small state/store module. No heavy framework — easiest for us to maintain together. Built as a PWA.
Local data
IndexedDB via Dexie.js for the catalog; OPFS (origin-private file system) for original photo bytes. This is the source of truth on each device — the app reads/writes here first, always.
Sync & backend
Supabase — one managed service giving you Postgres (catalog), Auth (accounts), Realtime (live sync), and Storage (thumbnails + optional originals). Generous free tier; no server for us to run.
Hosting
Static app on Cloudflare Pages or Netlify — free, automatic HTTPS, global CDN, deploys on git push. Required for PWA install & service workers.

Why Supabase over rolling our own

Auth, database, realtime and file storage in one dashboard means far less for a non-developer to wire up or babysit. If we ever outgrow it, the data is plain Postgres — fully portable.

Simpler free alternative

If you'd rather have more storage headroom, Cloudflare D1 + R2 (10 GB free, no download fees) is the cost-optimal swap for the backend — at the price of a bit more wiring. Supabase is the recommended default for ease.

Two columns make sync work everywhere

Every synced row carries updated_at and a deleted tombstone flag. That's all the sync engine needs to figure out what changed since last time and to propagate deletes — no fragile diffing. content_hash lets two devices recognise the same original and de-duplicate.

Free-ish

Point Lumen at a folder in Google Drive / iCloud / Dropbox you already pay for. Lumen references the files; you reuse storage you own. No new bill.

Cloudflare R2 / Backblaze B2 at roughly $1.50–6/mo for 100–400 GB, no download fees. Every original on every device, automatically. Add this only if you want zero gaps.

Bottom line

The full workflow — import, analyze, organize, edit, sync, export — runs completely free. The only thing that ever costs money is choosing to keep a second copy of the raw originals in the cloud, and that's a switch you can flip whenever you want.

05

Authentication

It's your personal library, so auth has one job: prove "this is me" on each device so they sync to the same account. Kept deliberately minimal.

How you sign in

Supabase Auth with a magic email link or "Sign in with Google." No passwords to manage. First launch on a new device: sign in once, and it starts pulling your catalog.

How your data stays yours

Postgres Row-Level Security ties every row to your user_id. Even though it's a shared service, the database enforces that only your account can read or write your photos.

Local works without login

Because the app is local-first, it stays fully usable on a device before you ever sign in — auth only switches on cross-device sync. You're never locked out of your own photos by a login screen.

06

Sync strategy

Local-first means the app never waits on the network. You always read and write the local database; sync happens quietly in the background and reconciles when there's a connection.

1 · Write local

Every edit lands in IndexedDB instantly with a fresh updated_at. The UI reacts immediately — zero latency.

2 · Queue change

The row id joins an outbox. Offline? It simply waits — nothing is lost.

3 · Push & pull

On reconnect: push outbox rows, pull anything on the server newer than your last cursor. Realtime makes this near-instant.

4 · Merge

Conflicts resolve by newest updated_at, field-by-field. Deletes ride along as tombstones.

Why conflicts are a non-issue here

You're one person rarely editing the same photo on two devices at the same second. And because edits are independent numeric fields, "newest wins per field" is both simple and correct. We can layer richer merging later, but we won't need it on day one.

07

Offline support & PWA

Offline isn't a feature bolted on — it's the default state of a local-first app. The PWA layer makes it installable and resilient on every device.

Installable

A manifest.json + icons make Lumen installable to the home screen / dock on iOS, Android, Windows and macOS — a real app window, no browser chrome, one codebase.

Service worker

Built with Workbox: caches the app shell so it launches offline, and serves thumbnails from cache for instant browsing on a plane or a weak hotel connection.

Background sync

The Background Sync API flushes your outbox automatically when connectivity returns — even if Lumen isn't open. Edit on the plane, land, and it syncs itself.

Native-feeling intake

File Handling + Web Share Target let you "Open with Lumen" or share photos straight from your phone's camera roll into the import pipeline.

08

Deployment

Two things to host, both on free tiers, both deploy by pushing to a git repo.

1

Repository & build

Put Lumen in a Git repo (GitHub). A light build step (Vite) bundles the app and generates the service worker. No build server to maintain.

2

App → Cloudflare Pages / Netlify

Connect the repo; every push auto-deploys to a global CDN with HTTPS and a custom domain (e.g. lumen.yourname.com). HTTPS is required for PWA + service workers.

3

Backend → Supabase project

One project holds the Postgres schema, Auth and Storage. Schema changes are tracked as SQL migration files in the same repo, so the database is reproducible and versioned.

4

Config & secrets

Only the public Supabase URL + anon key ship to the client; Row-Level Security does the protecting. Nothing sensitive lives in the front-end.
