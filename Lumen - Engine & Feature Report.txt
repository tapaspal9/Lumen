PART 1 — WHAT LUMEN IS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Lumen is a privacy-first, browser-based travel photo editor. It runs entirely
in the browser — no server, no uploads, no accounts required. The imaging
engine operates directly on raw pixel buffers (ImageData / Uint8ClampedArray),
which makes it fast, framwork-free, and fully offline-capable.

The current codebase is a mature prototype. The core engine is described
internally as "sacred" — meaning it is intentionally isolated from storage,
sync, and UI concerns so it can evolve independently. The architecture
separates the codebase into three layers: UI, a central state store, and a
swappable storage provider interface that is cloud-ready.

Lumen is a PWA (Progressive Web App): it ships a manifest.json, service
worker, and Apple/Android meta tags so it can be installed to any device's
home screen or dock and launched like a native app.

PART 2 — THE ENGINE IN DETAIL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The engine lives in five core JS files. Each is a self-contained IIFE module
that exposes a global object (Imaging, Analysis, Presets, Scene, Suggestions).
None of them touch the DOM or any storage API. They are pure functions that
accept data and return data.

────────────────────────────────────────────────────────────────────
2.1  imaging.js — Pixel processing pipeline
────────────────────────────────────────────────────────────────────

This is the heart of the engine. It exposes:

  Imaging.analyze(imageData)
    A full single-pass pixel analysis. Builds per-channel histograms (R, G, B,
    luminance), computes mean and standard deviation of luminance, mean RGB
    per-channel, per-pixel saturation, shadow and highlight clipping ratios,
    black/white point percentiles, a 4×4×4 RGB color cube (64 buckets) for
    dominant color extraction, an edge-energy map via sampled gradient
    magnitude, spatial region brightness (top/bottom/left/right/center), and
    48-bucket horizontal and vertical luma + edge profiles for composition
    analysis. All of this runs in one loop over the pixel data.

  Imaging.quickHist(imageData)
    A lightweight histogram-only pass used for live rail updates as the user
    drags sliders — avoids the cost of the full analysis.

  Imaging.process(src, params, out)
    The main adjustment pipeline. Applies all slider values to a source
    ImageData buffer and writes results into an output buffer. The processing
    order is intentional:
      1. White balance  — per-channel gain (temperature adjusts R/B ratio,
                          tint adjusts G).
      2. Exposure       — global multiplicative gain via 2^(ev × 1.6).
      3. Tone curves    — blacks, whites, shadows, highlights applied via a
                          smoothstep-based zone-weighting system (broad shadow
                          and highlight zones, deep black/bright white zones).
      4. Contrast       — linear pivot around 0.5.
      5. Clarity        — large-radius unsharp mask (radius ≈ min(w,h)/48),
                          enhances midtone local contrast.
      6. Sharpness      — small-radius (1px) unsharp mask.
      7. Noise          — small-radius blur blend (smoothing).
      8. Saturation     — scales chroma relative to luminance.
      9. Vibrance       — saturation boost that is inversely weighted by
                          existing saturation (protects already-saturated
                          pixels, lifts dull ones).
     10. Black & white  — linear blend toward luminance gray.
     11. Vignette       — radial distance-based brightness modifier.

    All intermediate computations work in float (0..1) range via Float32Array
    buffers. The final step clamps and packs back to 0..255 in the output
    ImageData. All adjustments are non-destructive — original pixel data is
    never modified.

  Imaging.straightenedCanvas(img, angleDeg, BW, BH)
    Renders a rotated + cover-scaled version of an image to a new canvas.
    Uses a computed scale factor (coverScale) so that rotating never reveals
    empty corners — identical to how Lightroom straighten works.

  Imaging.DEFAULTS
    The zero-point parameter object for all 15 sliders:
    exposure, contrast, highlights, shadows, whites, blacks,
    temperature, tint, saturation, vibrance, sharpness, clarity,
    noise, vignette, bw.

────────────────────────────────────────────────────────────────────
2.2  analysis.js — Interpretation & auto-correction brain
────────────────────────────────────────────────────────────────────

Interprets raw pixel statistics into scores, detected issues, correction
parameters, and plain-English explanations.

  Analysis.exposureScore(stats)
    Scores 0–100. Penalises deviation from a target mid-brightness (~47%),
    shadow/highlight clipping, and low dynamic range (narrow histogram).

  Analysis.colorHarmonyScore(stats)
    Scores 0–100. Rewards saturation near an ideal band, colour variety in
    dominant palette, penalises color cast magnitude.

  Analysis.colorCast(stats)
    Measures the RB axis (warm/cool) and the green axis (green/magenta) to
    detect and label casts: "Warm cast", "Cool / blue cast", "Green cast",
    "Magenta cast". Returns vector magnitude so severity can be graded.

  Analysis.detectIssues(stats)
    Returns a labelled list of detected problems with severity 0/1/2:
    Underexposed, Overexposed, Flat/low contrast, Dull undersaturated color,
    Crushed shadows, Blown highlights, color cast, Hazy/lifted blacks.
    If nothing is wrong, returns "Well balanced — minor polish only."

  Analysis.autoParams(stats, strengthKey)
    The auto-correction formula. Computes all 15 slider values from the pixel
    statistics. Takes a strength key ('subtle', 'natural', 'professional',
    'dramatic') which multiplies output by 0.45, 0.72, 1.0, or 1.45
    respectively. Handles: exposure from mean brightness deviation, contrast
    from histogram sigma, black/white point expansion, shadow/highlight
    recovery from clipping ratios, vibrance from saturation deficit, clarity
    and sharpness as constants scaled by strength, noise reduction at low
    strength, vignette at dramatic/professional levels. All values are clamped
    to –100…+100.

  Analysis.autoParamsPortrait(stats, strengthKey)
    A separate formula tuned for portrait/skin subjects. Uses 55% of the
    effective strength, greatly reduces clarity (to 2×k vs 9×k), reduces
    sharpness, saturation, and contrast to protect skin texture and tones. The
    white balance correction is also more conservative (75% of strength).

  Analysis.autoParamsForScene(stats, strengthKey, scene)
    Dispatcher: routes to autoParamsPortrait when scene.isPortrait is true,
    otherwise to autoParams. This is the correct function to call from the UI.

  Analysis.suggestCrop(stats, aspectRatio, imageAspect)
    Heuristic composition crop assistant. Reads the 48-bucket vertical and
    horizontal luma and edge profiles to:
      - Detect a horizon line (strongest luminance step in the vertical
        profile). If the step is dominant (>1.7× average), the horizon is
        aligned to the nearest rule-of-thirds line.
      - Find the subject centroid (weighted center of edge energy in both
        axes) and place it on the nearest rule-of-thirds intersection.
      - Scale the crop frame to the target aspect ratio within an 88% safe
        box (gentle trim of distracting edges).
    Returns a normalized {x, y, w, h, angle} crop rectangle.

  Analysis.explain(stats, params)
    Generates human-readable sentences explaining what was changed and why,
    based on the measured statistics and the applied parameters.

  Analysis.cropRationale(cropResult, aspectLabel)
    Generates sentences explaining the crop suggestion.

────────────────────────────────────────────────────────────────────
2.3  presets.js — Preset library
────────────────────────────────────────────────────────────────────

38 original creative presets across 6 categories:
  Landscape (8)        — Natural, Vibrant, Cinematic, Dramatic Sky,
                         Deep Greens, Mountain Light, Forest Mist, Ocean & Coast
  Travel (6)           — Documentary, Warm Vacation, Luxury, Adventure,
                         Film Travel, Golden Road
  Portrait (6)         — Natural, Warm Skin Tones, Editorial, Soft,
                         Golden Hour Portrait, B&W Portrait
  City & Architecture  — Urban, Street Photography, Clean Architecture,
  (5)                    Night City, Concrete & Steel
  Food (4)             — Restaurant, Bright & Fresh, Moody Food, Golden Plate
  Creative (9)         — Film Look, Matte, Vintage, Moody, High Contrast,
                         Golden Hour, Nordic Cool, Desert Heat, B&W Dramatic
  Plus 5 backward-compatible alias presets.

Each preset stores:
  - A category and display name
  - A description string
  - A 3-colour swatch array for UI rendering
  - A "creative delta" object: relative adjustments to mix on top of the
    auto-correction base
  - Optional flags: portraitSafe (use portrait auto-base), mono (B&W preset)

  Presets.build(id, stats, intensity, scene)
    Mixes the auto-corrective base (full analysis output) with the preset's
    creative delta, then scales the merged result by intensity (0..1). Portrait
    presets automatically use the portrait-safe auto base. This is what gets
    sent to Imaging.process().

  Presets.recommendFor(sceneType)
    Returns an ordered list of 4 preset IDs for a given detected scene type,
    using a curated scene→preset map.

────────────────────────────────────────────────────────────────────
2.4  scene.js — Smart scene classifier
────────────────────────────────────────────────────────────────────

Heuristic scene detection — no ML model. Runs entirely on the pixel
statistics already computed by Imaging.analyze(). Classifies into 10 types:
  Sunset / Golden Hour, Landscape, Nature, Beach & Water, City & Architecture,
  Food, Portrait, Night, Low Light, Indoor.

Each type has an independent scoring formula combining:
  - Color family weights from the dominant-color palette (blue, green, warm,
    neutral, skin proportions)
  - Region brightness relationships (topBright, centerBright)
  - Edge energy (high edge → structured scene, low edge → open vista)
  - Derived flags: isNight, isLowLight, hasPortraitSkin, isIndoor

Skin detection is built into the color family classifier — checks warm
mid-range RGB ratios (r > g > b, r 110–245, specific delta ranges) rather
than requiring a face-detection model.

Confidence is calculated from the score gap between the top two candidates,
producing a value in the range 0.38–0.97. The result also includes:
  - isPortrait flag (used to route to portrait-safe corrections)
  - hasFaces flag (skin colour detected in center)
  - isNight / isLowLight flags
  - Top 3 candidates (used in any confirm dialog)

Scene.advice(scene) returns a one-sentence editing recommendation per scene.

────────────────────────────────────────────────────────────────────
2.5  suggestions.js — AI suggestion engine
────────────────────────────────────────────────────────────────────

Turns measured statistics into a ranked list of optional, user-approvable
suggestions. Each suggestion has an id, icon, tag, title, detail sentence,
and an action (either a delta merge of specific sliders, or a preset apply).

Suggestion types generated:
  - Scene detection result (if confidence ≥ threshold)
  - Exposure correction
  - Contrast / clarity boost
  - White balance correction
  - Colour richness / vibrance boost
  - Composition / crop advisory (triggered by strong vertical brightness skew)
  - Mood preset recommendation
  - Black & white suitability (scored from contrast, texture, tonal range,
    and inverse of colour richness)

Suggestions.aiNote(stats) is an async function that, when a claude.complete()
interface is available in the window, calls Claude via a structured prompt
containing pixel statistics and asks for a 2–3 sentence photography editor's
note. This is the only external call in the entire engine.

────────────────────────────────────────────────────────────────────
2.6  exif.js — EXIF reader + GPS privacy
────────────────────────────────────────────────────────────────────

A hand-written JPEG/EXIF parser (no dependencies). Reads the APP1/TIFF block
directly from an ArrayBuffer. Extracts: camera Make, Model, LensModel, ISO,
ExposureTime, FNumber, FocalLength, DateTime. Detects GPS presence and stores
the GPS IFD pointer offset.

Privacy operations:
  - Exif.stripGPS(app1, gpsOff) — zeroes the GPS IFD pointer so no location
    data is accessible from the output file.
  - Exif.inject(blob, app1) — re-inserts the (optionally scrubbed) EXIF block
    into a canvas-exported JPEG, since canvas toBlob strips all metadata.

────────────────────────────────────────────────────────────────────
2.7  Supporting engine files
────────────────────────────────────────────────────────────────────

  histogram.js   — Renders live RGB and luminance histograms to canvas elements
                   in the right panel. Redraws on every slider change.

  crop.js        — Full crop & straighten overlay UI. Handles all 6 resize
                   handles (N/S/E/W/NW/NE/SW/SE), drag-move, ratio locking
                   (Original / 16:9 / 4:3 / 3:2 / 1:1 / Custom), straighten
                   slider, auto-crop (calls Analysis.suggestCrop), and preview
                   mode. Applies the crop by rendering a new canvas at the
                   cropped region dimensions.

  export.js      — Web export modal. Configurable output size, quality,
                   format (JPEG/WebP), GPS strip toggle, filename. Re-runs
                   Imaging.process at export-time on the original (lossless
                   pipeline), re-injects EXIF via Exif.inject, offers download.

  panels.js      — Right-rail UI: renders sliders, scores, issue badges, white
                   balance swatch, dominant colour swatches, preset grid,
                   EXIF metadata panel, and the "What Lumen changed" explanation
                   block. All controls update the current photo entry and
                   trigger a re-render pass.

  dashboard.js   — Gallery / dashboard view: photo grid, batch auto-enhance
                   (applies autoParams to every photo in the library), import
                   button.

PART 4 — WHAT CAN BE IMPROVED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

These are grouped from highest-impact / most foundational to more incremental.

────────────────────────────────────────────────────────────────────
4.1  CRITICAL: Library data does not survive a page refresh
────────────────────────────────────────────────────────────────────
The store + provider infrastructure exists in js/core/ but app.js has not yet
been wired to call Store.addPhoto(), Store.saveEdit(), or to re-hydrate the
filmstrip from IndexedDB on load. The library currently lives only in memory.
This is the single most important gap — without it, every session starts from
scratch. This is Phase 1 in the technical roadmap and should be the very next
thing built.

────────────────────────────────────────────────────────────────────
4.2  Processing runs on the main thread (blocks UI)
────────────────────────────────────────────────────────────────────
Imaging.process() is a synchronous pixel loop. On a large photo (e.g. a
24 MP JPEG) this can block the main thread for 200–600ms, freezing the
entire UI. The fix is to move imaging into a Web Worker or OffscreenCanvas
worker. The engine's pure-function design (no DOM, no storage calls) makes
this a clean migration — it just needs postMessage to pass ImageData in/out.
This would also enable non-blocking live preview during slider drags.

────────────────────────────────────────────────────────────────────
4.3  Histogram color analysis is coarse (4-bit color cube)
────────────────────────────────────────────────────────────────────
The dominant color extraction uses a 4×4×4 cube (64 buckets), meaning each
bucket spans a 64-value range in each channel. This is too coarse to
distinguish, for example, a teal sky from a blue sea, or a green forest from
an olive field. Increasing to an 8×8×8 cube (512 buckets) would give much more
accurate dominant colors, better scene classification, and better preset
recommendations — at a modest memory cost (512 × 4 bytes vs 64 × 4 bytes).

────────────────────────────────────────────────────────────────────
4.4  Scene classifier has no feedback / correction loop
────────────────────────────────────────────────────────────────────
The scene classifier is fully heuristic and can misfire (e.g. classifying a
food photo with warm tones as "Sunset", or a portrait with a blue background
as "Beach & Water"). There is no mechanism for the user to correct it and no
learning from corrections. Adding a simple "Not [Scene type]" button on the
scene suggestion card would let users reject misclassifications, and those
corrections could be stored and used to adjust thresholds over time.

────────────────────────────────────────────────────────────────────
4.5  No thumbnail generation pipeline
────────────────────────────────────────────────────────────────────
The filmstrip and dashboard grid render from the original full-resolution
image objects. As the library grows, this will consume large amounts of memory
and slow down the gallery view significantly. A proper import pipeline should
generate a small JPEG thumbnail (e.g. 240px wide) per photo at import time,
store it in the blobs IndexedDB store, and use that for all gallery/filmstrip
rendering. The full image should only be decoded when the photo is opened for
editing.

────────────────────────────────────────────────────────────────────
4.6  Auto-enhance formula uses fixed targets, not perceptual goals
────────────────────────────────────────────────────────────────────
The auto-correction in Analysis.autoParams() drives toward fixed numeric
targets (mean luminance ~47%, saturation ~42%, etc.). These are reasonable
averages for travel photography but they don't adapt to the subject or intent.
A forest mist photo and a bright beach scene should have different target
exposure levels. The scene classifier already runs before auto-enhance — it
should feed back into the target values so each scene type has its own ideal
tonal goal.

────────────────────────────────────────────────────────────────────
4.7  Colour tone curves are channel-independent
────────────────────────────────────────────────────────────────────
The tone() function applies the same blacks/whites/shadows/highlights to R, G,
and B equally. This means a warm photo will shift warmer in the highlights and
cooler in the shadows, etc. A real tone-curve system separates the channels so
highlight tint and shadow tint can be independently controlled (the classic
teal/orange split-tone effect). Adding per-channel tone adjustments would open
up a significant class of cinematic grades currently only available via the
preset system.

────────────────────────────────────────────────────────────────────
4.8  No RAW file support
────────────────────────────────────────────────────────────────────
The engine ingests JPEG and to a lesser extent HEIC (where the browser decodes
it). RAW formats (CR3, NEF, ARW, DNG) are not supported — the browser cannot
decode them natively. Adding RAW support would require integrating a WASM-based
RAW decoder (e.g. libraw or dcraw compiled to WASM). This is non-trivial but
would dramatically expand the target audience (any serious travel photographer
shoots RAW).

────────────────────────────────────────────────────────────────────
4.9  Clarity and sharpness use the same algorithm (unsharp mask)
────────────────────────────────────────────────────────────────────
Both clarity and sharpness are implemented as unsharp masks — the difference
is only the blur radius (large for clarity, 1px for sharpness). Real clarity
is a midtone-only contrast boost: it should not affect highlights or shadows.
Adding a midtone mask (weighing the clarity boost toward the 0.2–0.8 luminance
range and fading it at extremes) would make the clarity control behave more
like the Lightroom version and reduce the risk of blowing highlights or
crushing shadows when used aggressively.

────────────────────────────────────────────────────────────────────
4.10  No masking or local adjustments
────────────────────────────────────────────────────────────────────
All adjustments are global. There is no way to selectively brighten a sky,
darken a foreground, enhance a subject, or apply different grades to different
areas. This is a significant gap vs. Lightroom-class tools. A minimal but
high-value first step would be a brush/gradient radial mask system with a
single set of local adjustments (exposure, saturation, clarity). The engine
is well-positioned for this since process() already operates pixel-by-pixel
and could accept a per-pixel weight map.

────────────────────────────────────────────────────────────────────
4.11  No cross-device sync (planned but not yet built)
────────────────────────────────────────────────────────────────────
The architecture is fully designed for catalog sync via Supabase and the
IStorageProvider pattern, but it is not implemented. Users on multiple devices
(phone, laptop, desktop) currently have completely separate libraries. Phase 4
of the roadmap covers this. Until then, edits made on one device are invisible
on another — a major friction point for a travel app.

────────────────────────────────────────────────────────────────────
4.12  Trip / album management is missing
────────────────────────────────────────────────────────────────────
The data model supports albums and album_photos tables, and the dashboard
exists, but there is no UI to create, name, or manage trip albums. All photos
currently sit in a single flat library. For a travel-focused product, grouping
photos by trip/destination is core functionality. This is Phase 2 of the
roadmap.

────────────────────────────────────────────────────────────────────
4.13  Noise reduction is actually a blur, not real noise reduction
────────────────────────────────────────────────────────────────────
The noise slider blends the image toward a 1-pixel Gaussian blur. This
reduces noise but also destroys fine detail uniformly. Real noise reduction
uses a luminance-only bilateral or guided filter approach (smooth chroma
independently from luma) so detail edges are preserved while smooth areas
(skies, skin) are denoised. Implementing this would require a more complex
filter but the benefit for ISO-pushed travel photos (interiors, evening
scenes) would be substantial.

────────────────────────────────────────────────────────────────────
4.14  Vignette formula is simplistic
────────────────────────────────────────────────────────────────────
The current vignette is a brightness multiplier based on distance from center,
applied in linear RGB. It can go both positive (brighten edges) and negative
(darken). A more refined vignette would: (a) apply in luminance rather than
linear RGB to avoid color shifts, (b) offer a separate feather/roundness
control, and (c) limit the vignette to the "post-highlights" stage to avoid
crushing already-dark edge areas.

────────────────────────────────────────────────────────────────────
4.15  No keyboard shortcut system
────────────────────────────────────────────────────────────────────
Only Cmd/Ctrl+Z for undo is implemented. A pro photographer workflow expects
shortcuts for: next/previous photo, auto-enhance, compare toggle, crop mode,
preset navigation (1–9 for top presets), export. The engine architecture
doesn't prevent this — it just hasn't been built.

────────────────────────────────────────────────────────────────────
4.16  Mobile editing experience is minimal
────────────────────────────────────────────────────────────────────
A mobile bottom-nav exists but the editing experience on touch screens is not
fully realised. Phase 3 of the roadmap calls for: pinch-zoom on the canvas,
touch-draggable crop handles, swipe gesture for before/after comparison, and
44px+ hit targets on all controls. Sliders in particular are difficult to use
with a thumb at their current density.

────────────────────────────────────────────────────────────────────
4.17  External CDN dependencies in production
────────────────────────────────────────────────────────────────────
React, ReactDOM, and Babel are currently loaded from unpkg.com CDN at runtime.
In production, these should be locally bundled to: (a) work fully offline,
(b) avoid a failed CDN load breaking the app, and (c) remove any reliance on
a third-party service. The roadmap calls for a Vite build step which would
address this, but it is not yet in place.
