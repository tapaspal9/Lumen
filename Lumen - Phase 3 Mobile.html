01

Mobile-first philosophy

Phase 3 is not a responsive rearrangement of the desktop UI. It is a reimagined editing experience designed around a thumb, a 6-inch screen, and a user who is standing in a train station in Tokyo.

Constraint 1

Thumb-first layout

Every interactive element is in the bottom two-thirds of the screen. No controls require a stretch to the top corner. Minimum touch target: 44×44pt everywhere.

Constraint 2

One hand, always

The full import → auto-enhance → export workflow can be completed one-handed. Sliders, presets, and the export button are all reachable without a second hand or repositioning the phone.

Constraint 3

Never block the view

Controls slide away when the user is reviewing the photo. The image fills the screen. Panels appear only when needed — as bottom sheets, not permanent sidebars.

Rule 2 compliance — no module imports another

The entire Phase 3 UI is wired through LumenBus. Touch gestures emit events (gesture:pinch, edit:params-changed). The Web Worker result fires imaging:preview-ready. M7 listens and updates the canvas. No new direct module references are introduced anywhere.

02

Performance architecture

The single biggest performance unlock is the preview pipeline: process an 800px-wide proxy image during editing, and only run the full-resolution pipeline on export. A 48MP photo (8000×6000) has 48 million pixels. An 800px preview has 640,000 — 75× fewer. Sliders become instant.

Importonce per photo
Read original file → decode → generate 800px preview ImageData → store both to M8. main thread Preview generation takes ~120ms on iPhone 14.
Slider / presetevery interaction
User moves slider → LumenBus.emit('edit:params-changed', params) → Web Worker receives preview ImageData + params → Imaging.process() on 640K pixels → posts result back → canvas updates. Web Worker ≈16ms target
Exporton demand only
User taps Export → Web Worker receives full-resolution ImageData + params → Imaging.process() → EXIF inject → share sheet. Web Worker Full 48MP render: ~4–6 seconds, UI fully responsive throughout.
Full-res decodewhat we avoid
The original full-resolution ImageData is decoded once at export time, not held in memory during editing. never during editing This caps working memory at ~30MB regardless of source file size.
// js/workers/imaging-worker.js
// The engine runs identically in a Worker — it has no DOM dependencies.
importScripts('../imaging.js');

self.onmessage = ({ data }) => {
  const { id, src, params } = data;
  const out = new ImageData(src.width, src.height);
  Imaging.process(src, params, out);
  // Transfer the buffer — zero copy, no clone
  self.postMessage({ id, out }, [out.data.buffer]);
};

// js/platform/mobile.js — wired by M7, uses LumenBus (Rule 2 ✓)
const worker = new Worker('js/workers/imaging-worker.js');
worker.onmessage = ({ data }) => LumenBus.emit('imaging:preview-ready', data.out);

LumenBus.on('edit:params-changed', (params) => {
  worker.postMessage({ id: 'preview', src: previewImageData, params },
    [previewImageData.data.buffer]); // transferable — main thread yields ownership
});
Memory budget

≤ 150 MB at any time

Preview ImageData: ~2 MB. Thumbnail: ~0.2 MB. Params + metadata: negligible. Full-res only allocated during export, then freed. iOS kills apps above ~200 MB.

GPU acceleration

CSS filters for histogram display

Histogram canvas uses requestAnimationFrame with willChange: transform. Preview canvas is composited on the GPU layer — no main-thread repaints during updates.

Large library

Virtualised grid

The library grid renders only visible thumbnails. Off-screen tiles are removed from the DOM. 10,000 photos load in under 300ms. Thumbnails are always 240px — never original size.

03

Mobile editing workflow

Seven steps, designed so the user can go from camera roll to shared photo in under 30 seconds without touching a single manual slider.

1

Import photo

Tap the + button → PHPickerViewController (Capacitor) or Files app sheet. Supports multi-select. Thumbnail and preview are generated in the background immediately after selection — import feels instant.

2

Quick analysis preview

Imaging.analyze() runs on the 800px preview in the Web Worker. Takes ~40ms on iPhone 14. Shows exposure, color, sharpness scores as animated score bars while the user is still looking at the photo.

3

Detect photo type

Scene.classify() runs in the same Worker pass. Result label appears as a subtle badge on the photo: "Landscape · 91% confident". Portrait mode routes to portrait-safe corrections automatically.

4

Suggest presets

Top 4 recommended presets appear as thumbnail swatches below the photo. Scene-matched via Presets.recommend(). User can see the effect instantly — tap a preset, preview updates in the Worker within one frame.

5

Apply edits

Tap Auto-Enhance or any preset swatch. The corrected preview appears immediately. A subtle "Before / After" swipe hint appears for 2 seconds then fades. No explicit confirmation needed.

6

Fine tune

Swipe up to reveal the adjustment panel — exposure, vibrance, contrast as large scrollable sliders. Each drag fires edit:params-changed; preview updates in real time at 60fps. Tap the canvas to dismiss.

7

Export / share

Tap the share button → native iOS share sheet via Capacitor. Full-resolution render runs in the Worker during the sheet's open animation (≈1.5s) — by the time the user picks an app, the file is ready.

30-second goal

Import → Auto-Enhance → Export in under 30 seconds on a 12MP iPhone photo. Steps 1–5 complete without a single manual slider interaction — the AI does the work.

Travel Mode shortcut

In Travel Mode, the workflow collapses to 4 steps: Import → Auto-Enhance → Preview → Share. Steps 3, 4, and 6 are skipped. Optimised for speed over perfectionism when you have 3% battery left.

Offline behaviour

All 7 steps work with no internet. AI analysis uses local heuristics. Presets are bundled. Export goes to camera roll. If Travel Mode queues an upload, it stores the export and fires it when Wi-Fi returns — the user is never blocked.

04

Touch interface

Every control is designed for touch as the primary input, not mouse interaction with enlarged targets. Gesture bindings are additive — they work alongside tap interactions, never replace them.

🤏

Pinch zoom

Two-finger pinch on canvas. Smooth scale from 1× to 4×. Two-finger pan when zoomed in. Double-tap resets to fit. Min touch target ignored for canvas — entire canvas area responds.

👆

Tap to compare

Single tap on the canvas toggles Before / After. Hold shows original; release shows edit. No button needed. A subtle "Before" badge appears on the first tap to educate new users.

↔️

Swipe before / after

Horizontal swipe on canvas when not zoomed reveals a split-view divider. Left = before, right = after. The divider snaps to the finger position. Swipe off-screen to dismiss.

⬆️

Swipe up — adjustments

Swipe up from bottom of canvas reveals the adjustment panel as a bottom sheet. Swipe down to dismiss. Sheet has three tabs: Adjustments · Presets · Analysis.

✂️

Gesture crop

In crop mode, drag corners and edges with enlarged 44pt hit areas. Two-finger rotate gesture maps to the straighten slider. Pinch inside the crop frame to rescale it.

◀▶

Swipe between photos

Horizontal swipe on canvas (when at 1× zoom) navigates to the previous or next photo in the library. Smooth page-curl transition. Current edit is auto-saved on swipe away.

📳

Haptic feedback

Light impact on: preset tap, crop handle snap to thirds, export complete, auto-enhance applied. Uses navigator.vibrate() on Android and the Capacitor Haptics plugin on iOS.

🎚️

Large sliders

All adjustment sliders are 56px tall with a 32px thumb. Dragging anywhere on the slider track (not just the thumb) responds. Labels show numeric value during drag. Tap the label to type a value.

No tiny buttons anywhere in Phase 3

Every tappable element must meet the 44×44pt minimum. Existing desktop UI elements (the score cards, the preset pills, the export config fields) are rebuilt as native-feeling mobile components — not CSS-scaled versions of the desktop elements.

07

Travel Mode

A one-tap mode for the moment you land at an airport with 12% battery and 40 photos to process. Travel Mode trades fine-grained control for speed and power efficiency.

Travel Mode

Phase 3 feature

Low battery

Preview resolution drops from 800px to 480px — 2.8× fewer pixels to process. Web Worker runs at reduced priority. Screen brightness locked to 60%. Estimated battery saving: ~35% vs normal editing session.

Speed over perfectionism

Auto-Enhance fires immediately on import. Manual sliders are hidden — only a single Intensity control (Subtle → Dramatic) is visible. The editing workflow collapses from 7 steps to 4. Batch enhance available with one tap.

Export queue

Exports are queued rather than run immediately. Full-resolution renders fire only when on Wi-Fi and charging. The queue shows in the Library as a pending badge. User can force-flush the queue manually at any time.

Activation: Settings → Travel Mode toggle, or auto-prompt when battery drops below 20%. State is persisted via Store.saveSetting('travelMode', true). Deactivates automatically when plugged in and idle for 2 minutes.

Offline presets — always available

All 38 built-in presets are bundled in presets.js — no network required. Travel Mode does not change this. Offline editing has always been a core principle.

Module boundary

Travel Mode is a UI state stored in M8 (Storage). M7 reads it on boot and adjusts the interface. No engine module knows Travel Mode exists. Turning it off restores full UI immediately.

08

Mobile storage

Mobile storage is deliberately conservative. The rule: store the smallest representation that enables the intended use case. Never store a second copy of original pixels.

What we store
Original blob photos.app ref or File
240px thumbnail ~30 KB JPEG
800px preview ~180 KB JPEG
Edit parameters ~1 KB JSON
Preset selection id + intensity
What we never duplicate

The original full-resolution image is only loaded from disk once — at export time. It is never stored a second time in IndexedDB or duplicated into working memory during editing.

A library of 500 iPhone photos uses ~110 MB of storage (thumbnails + previews + params). The originals stay in the Photos app — Lumen holds a reference, not a copy.

10

Module impact

Phase 3 touches five of the eight modules. Three modules — including the sacred imaging engine — are completely unchanged. Every cross-module call is via LumenBus.

ModuleImpactWhat changes
M1 · Library Minor Add preview generation — at import time, generate and store an 800px preview JPEG alongside the thumbnail. importFiles() gains a generatePreview() step. Virtualised grid rendering for large libraries.
M2 · Imaging None Zero changes. Imaging.process() runs identically in the Web Worker. The engine doesn't know or care that it's in a worker thread — that's the payoff of Rule 1.
M3 · AI Engine None Zero changes. Analysis.autoParams(), Scene.classify(), and Suggestions all run in the same Worker thread as before. No interface changes required.
M4 · Presets None Zero changes. Preset swatch thumbnails are generated by passing preview ImageData through Presets.build() and Imaging.process() in the Worker — no new API needed.
M5 · Crop Minor Touch handle enlargement — crop handle hit areas expand to 44pt. Straighten slider maps to two-finger rotate gesture via M7. The CropEngine interface is unchanged; only the UI overlay in M7 is updated.
M6 · Export Minor Share sheet destination — register a new IExportDestination that calls the Capacitor Share plugin. Travel Mode queue added: renderBatch() now supports deferred execution via a defer: true option.
M7 · UI Major Complete mobile UI rebuild — iPhone and iPad layouts, all gesture handlers, bottom sheets, Travel Mode toggle, large sliders, virtualised library grid, keyboard shortcuts for iPad. This is the primary Phase 3 workstream.
M8 · Storage Minor Preview cache store — add a previews object store to IndexedDB alongside blobs. IStorageProvider gains two methods: storePreview(photoId, blob) and loadPreview(photoId). Interface is additive — backwards compatible.

macOS version is unaffected

The Web Worker thread and preview pipeline are additive changes. The desktop app already works at full resolution — it simply gains the option of using preview-mode processing for live slider feedback, making the macOS experience faster too. Phase 3 is a net upgrade to all platforms.
