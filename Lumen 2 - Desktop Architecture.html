01

Four architectural rules

Every decision in Lumen 2 follows from these four rules. When an upgrade feels complicated, it is usually because one of these is being violated.

Rule 1

The engine is sacred

imaging.js, analysis.js, scene.js, presets.js — pure functions that accept data and return data. They know nothing about the DOM, storage, or any platform API. This constraint never changes.

Rule 2

Modules don't import each other

No module imports or requires another module's internals. All cross-module communication flows through LumenBus. This is what makes each module independently replaceable.

Rule 3

Everything goes through a provider

No UI or engine code touches IndexedDB, SQLite, iCloud, or any storage API directly. All persistence flows through IStorageProvider. Switching backends is one constructor call in app.js.

Rule 4

Platforms are a shell concern

macOS menus, Touch Bar, iOS gestures — all wired at the platform shell level only. The 8 modules are identical JavaScript on macOS, iOS, and the web. The shell adapts them; the modules never know which platform they're on.

03

Tauri platform shell

Tauri v2 wraps Lumen's existing web layer in a native macOS binary. The entire JS engine is untouched. Tauri adds the OS-level capabilities a professional photo app requires that the browser alone cannot provide.

File system

Real disk access

Read any folder, watch directories for new imports, write exports anywhere. No file pickers needed for watch-folder or batch workflows.

App size

~12 MB binary

Ships with system WebKit — no bundled Chromium. The final .app is 10–20× smaller than an equivalent Electron build.

Distribution

App Store + direct

Produces a signed .app and .dmg. Submit to the Mac App Store or distribute directly with notarization.

Zero engine rewrites

Tauri calls imaging.js, analysis.js, and every other engine module through exactly the same function signatures they already expose. A Rust backend handles OS calls; the JS layer is completely unchanged.

// js/platform/tauri.js — the ONLY new platform-specific file
// Nothing below touches the imaging engine.

import { open } from '@tauri-apps/plugin-dialog';
import { watchImmediate } from '@tauri-apps/plugin-fs';
import { SQLiteProvider } from './sqlite-provider';

// SQLiteProvider replaces IndexedDB — same IStorageProvider interface
window.Store = new LumenStore(new SQLiteProvider());
await window.Store.init();

// Watch folder — emits via LumenBus; Library module handles the rest
async function watchFolder(path) {
  return watchImmediate(path, (e) => {
    if (e.type === 'create') LumenBus.emit('folder:new-file', e.path);
  });
}

// Native menu wired here — calls LumenBus, no module importing required
await Menu.new({ items: [
  { id: 'import', text: 'Import Photos…', action: () => LumenBus.emit('ui:import') },
  { id: 'export', text: 'Export…',        action: () => LumenBus.emit('ui:export') },
]}).setAsAppMenu();

05

Event bus

LumenBus is a lightweight pub/sub object shared across all modules. Emitting on a channel requires no knowledge of who is listening — this is the structural guarantee that keeps modules independent.

ChannelPublished bySubscribersPayload type
photo:importedLibraryUI, AI EnginePhoto
photo:selectedUIUI, AI Engine, PresetsphotoId
edit:committedUIStorage, HistogramEditRecord
edit:resetUIUIphotoId
analysis:readyAI EngineUI{ stats, scene, suggestions }
export:started / :doneExportUI{ photoId, options? / blob }
plugin:installedPlugin ManagerPresets, ExportPluginManifest
storage:syncedStorageUI{ count, lastSync }
album:createdLibraryUI, StorageAlbum
 
// js/core/bus.js — the entire implementation
const LumenBus = (() => {
  const listeners = new Map();
  return {
    on(channel, fn) {
      if (!listeners.has(channel)) listeners.set(channel, new Set());
      listeners.get(channel).add(fn);
      return () => listeners.get(channel).delete(fn); // returns unsubscribe fn
    },
    emit(channel, payload) {
      listeners.get(channel)?.forEach(fn => fn(payload));
    },
  };
})();
window.LumenBus = LumenBus;
06

Storage evolution

Each phase adds a new provider. No existing code changes when a new provider is introduced — only the constructor call in app.js (or tauri.js) is updated.

Phase 1–2 · LocalProvider

now
Web: IndexedDB via existing LumenDB helpers. Desktop (Tauri): SQLiteProvider using tauri-plugin-sql — same IStorageProvider interface, ~5× faster reads, no 2 GB quota. Original image blobs are stored locally and never synced without an explicit user action.

Phase 3 · iCloudProvider

after iOS launch
Implements IStorageProvider using CloudKit JS + NSUbiquitousKeyValueStore for settings. Photo metadata and edits sync across Mac and iPhone/iPad. Original RAW/JPEG files sync via iCloud Drive only when the user explicitly requests it ("Download Original"). Conflict resolution: last-write-wins on edits, with a per-photo updatedAt timestamp.

Phase 4 · SupabaseProvider

multi-platform, cross-account
Implements IStorageProvider using the Supabase JS client. Enables sharing albums across accounts, team collaboration, and Android/Windows support (which have no iCloud). The sync cursor + outbox pattern tracks pending changes for background flush. Original blobs remain on-device; only thumbnails and edit records travel to the cloud.

Privacy guarantee holds across all phases

Original image pixels never leave the device without an explicit export action — this is structural, not a policy. storeOriginal() and loadOriginal() always point to local storage regardless of which provider is active.

07

Plugin system

Plugins are manifest-driven bundles. Three plugin types map to the three most natural extension points: preset packs, AI model replacements, and export destinations. Each type implements a declared interface.

Type: presets

Preset Packs

A presets.json array of preset objects. Installed via PresetEngine.installPack(manifest). Displayed in the existing preset grid under a pack label. Uninstallable without restart.

Type: ai-engine

AI Model

An ONNX or CoreML model bundle with a thin JS wrapper that implements the IAIEngine interface. Replaces classify() and/or autoParams(). The rest of the app calls it identically.

Type: export-dest

Export Destination

Implements IExportDestination: { id, label, icon, send(blob, meta) }. Registered via ExportEngine.registerDestination(). Appears in the export modal destination list immediately.

// Plugin manifest — lumen-plugin.json
{
  "id":          "lumen-moody-film",
  "name":        "Moody Film Collection",
  "version":     "1.2.0",
  "type":        "presets",            // "presets" | "ai-engine" | "export-dest"
  "entry":       "presets.json",       // main entry file
  "author":      "Studio Noir",
  "requires":    ">=2.0.0",
  "description":"12 cinematic film presets with analog grain simulation"
}

// Install flow (same for all plugin types)
const manifest = await loadJSON('lumen-plugin.json');
LumenBus.emit('plugin:installed', manifest);
// → PresetEngine, ExportEngine, and AIEngine each listen
// → Only the module matching manifest.type acts on it
08

iOS / iPadOS path

The recommended approach is Capacitor — it wraps the existing web app in a WKWebView and provides native plugins as a thin bridge. The entire imaging engine runs identically on iOS.

What Capacitor provides
  • 📁 File system — Camera Roll, Files app, iCloud Drive
  • 📤 Share sheet — native iOS share extension
  • 🖼️ Photo picker — PHPickerViewController
  • 💾 Secure storage — Keychain for credentials
  • 🔔 Haptic feedback — on crop handle snap, on export complete
UI Module — iOS variant
  • Tab bar: Library · Edit · Adjust
  • Stage: pinch-zoom + swipe before/after
  • Sliders: 44px touch targets, thumb-reachable layout
  • Crop handles: enlarged hit areas, haptic snap to thirds
  • Export: native share sheet via Capacitor plugin

100% of the JS engine runs unchanged on iOS

Imaging.process(), Analysis.autoParams(), every preset, every suggestion — identical. Capacitor's WKWebView supports ImageData, OffscreenCanvas, and IndexedDB. The only new code is the Capacitor plugin bridge wired in js/platform/capacitor.js — the same pattern as js/platform/tauri.js.

The imaging engine is never in the "changed" column

Every upgrade in this matrix either adds an optional parameter to imaging.js or touches a completely different module. The core pixel pipeline — 15 years of photographic tuning — is never at risk.
